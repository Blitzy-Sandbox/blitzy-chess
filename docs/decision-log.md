# Decision Log

This log records the non-trivial architectural choices and explicit deviations behind `blitzy-chess`. Per the Explainability rule, the rationale for a design choice lives here, not in code comments; see [`traceability-matrix.md`](traceability-matrix.md) for the requirement-to-file coverage.

| Decision | Alternatives | Rationale | Risk |
|----------|--------------|-----------|------|
| Backend on `FastAPI` + `Uvicorn` (ASGI) with WebSocket transport for game play. | `Flask`/WSGI serving synchronous HTTP/JSON (the earlier minimal spec). | Real-time multiplayer, streamed AI-thinking updates, and a non-blocking server need ASGI + WebSocket; the prompt supersedes the earlier Flask/WSGI interpretation. **(deviation)** | Async complexity; mitigated by offloading the search via `asyncio.to_thread()`. |
| `python-chess` enforces every rule on the server; the client `chess.js` board is display-only. | A custom hand-written rules engine; trusting client-side moves. | `python-chess` is battle-tested for legality, SAN, FEN, draw detection, and Zobrist hashing, and server authority prevents tampering. **(deviation)** | Third-party dependency; mitigated by pinning `chess>=1.10`. |
| Depend on `chess>=1.10`; code does `import chess`. | Use the `python-chess` distribution name instead. | On PyPI, `python-chess` is a thin stub that only depends on `chess`, so naming it directly is misleading. | Name confusion; mitigated by a note in `docs/onboarding.md`. |
| `backend/chess_ai/engine/` imports only `chess` and the standard library; no `FastAPI`, `Starlette`, or WebSocket code. | Let the engine reach into request or connection objects. | Purity lets the search run in a worker thread and be unit-tested in isolation. | A little extra plumbing to pass data in; acceptable. |
| WebSocket handlers offload the synchronous search with `asyncio.to_thread()`. | Run the search on the event loop; rewrite the search as async. | CPU-bound search would stall the event loop and every connection; threads keep the loop responsive while the search stays synchronous and simple. | The GIL limits CPU parallelism; acceptable for per-game search. |
| Key the transposition table on the 64-bit Zobrist value returned by `chess.polyglot.zobrist_hash(board)`; cap it at 256 MB / 2^20 entries. This corrects the earlier `board.zobrist_hash()` wording: python-chess 1.11.2 exposes no `Board.zobrist_hash()` method, so the Zobrist key is the module-level `chess.polyglot.zobrist_hash(board)`. **(deviation)** | The internal `board._transposition_key()`, which returns a tuple, or the non-existent `board.zobrist_hash()`. | A 64-bit integer key is compact and fast; the internal tuple key is unsuitable for a fixed-size table, and `board.zobrist_hash()` is not part of the python-chess API. | Hash collisions (negligible at 64 bits); memory bounded by the cap. |
| Pin `react-chessboard@^4`. | The 5.x line with its single `options`-object API. | 5.x replaced the individual props with one options object; the `frontend/src/components/GameBoard.tsx` wrapper uses the 4.x individual-prop API. | Misses 5.x features; revisit if the wrapper is rewritten. |
| Pin `tailwindcss@^3`. | The 4.x line, which is CSS-first and drops `tailwind.config.js`. | The board palette and theme are configured in `frontend/tailwind.config.js`, which the 4.x CSS-first model removes. | Misses 4.x; revisit if migrating the theme to CSS-first. |
| Pin `react` / `react-dom@^18` (and `@types/react` / `@types/react-dom@^18`). | React 19, the current line. | The prompt's floor is React 18; pinning keeps the test and type toolchain on a known-good line. | Misses React 19; low, since 18 is fully supported. |
| Hold game and room state in memory; only self-play recordings persist, as files in `backend/games/`. | A database or ORM for durable games and users. | The prompt specifies in-process state, and no persistence layer is in scope. | State is lost on restart; acceptable for anonymous play. |
| Ship observability now: structured logging with correlation IDs, tracing, a `/metrics` endpoint, health and readiness checks, and a dashboard template. | Defer observability to a later revision (the earlier spec). | The Observability project rule mandates it in the first release. **(deviation)** | Extra dependencies, and `/metrics` is unauthenticated (see the next row). |
| Expose `/metrics` without authentication. | Protect it behind auth or secrets. | Scope is local verification; production hardening is out of scope. | Would expose metrics if deployed as-is; recorded as out-of-scope future work. |
| Keep `backend/chess_ai/rooms/protocol.py` and `frontend/src/types/index.ts` in step by hand. | Generate the types from a shared schema. | The message set is small and stable, so code generation is overkill for this scope. | Drift if edited on one side only; mitigated by documenting the pairing. |
| Create `blitzy-deck/references/blitzy-reveal-theme.css` from the theme spec embedded in the Executive Presentation rule. | Omit the file or link an external theme. | The file is referenced but absent from the repo, and the rule embeds the full spec, so the build creates it. | Must match the embedded spec; low. |
| Route the five screens with a `useState` view machine in `frontend/src/App.tsx`; add no router library. | A router such as `react-router`. | Five flat screens need no nested routes or URL history, so a view-state switch is simpler. | No deep-linkable URLs; acceptable for this scope. |
| Build and serve the frontend with `Vite`; its dev server proxies `/ws/` and `/api/` to `localhost:8000`. | Create React App or a hand-rolled Webpack setup. | `Vite` gives fast dev startup and a simple proxy that gives the browser one origin for both WebSocket and REST. | None significant; `Vite` is the modern default. |
| Run the frontend build and test stack on `vite@8`, `vitest@4`, and `@vitejs/plugin-react@6`, above the prompt's `vite ^5` / `vitest ^1` floors. | Hold the pinned `vite ^5` / `vitest ^1` floors. | The pinned floors resolve to versions with known advisories (a critical Vitest advisory plus moderate Vite and esbuild advisories), and a release must pass `npm audit` with zero vulnerabilities. The patched line (`vite >= 6.4.2`, `vitest >= 4.1.0`, `esbuild >= 0.25`) is met by `vite@8` and `vitest@4`, which also lets `@vitejs/plugin-react` move to the prompt's intended `6.0.2`. `react@18`, `react-chessboard@4`, and `tailwindcss@3` stay pinned and unchanged. **(deviation)** | A major-version jump on the build and test tools; mitigated by a regenerated `package-lock.json` and green `tsc -b`, `eslint`, and `npm audit` checks. |
| Lint with an ESLint 9 flat config composing `typescript-eslint`, `eslint-plugin-react-hooks`, and `eslint-plugin-react-refresh`, and declare the browser globals inline. | Add the full `eslint-plugin-react` preset, or omit React rules entirely (the prior scaffold did the latter). | The Rules of Hooks and the fast-refresh export check are the load-bearing React 18 lint rules for this SPA; the broader `eslint-plugin-react` preset largely duplicates what TypeScript and the `react-jsx` transform already enforce. The few browser identifiers the SPA uses are declared inline rather than adding the `globals` package. | Misses some stylistic React rules; revisit if the component surface grows. |
| Derive the captured-pieces and material differential in `App.tsx` by replaying the server's SAN `move_history` through a throwaway `chess.js` instance, then pass the result to `SidePanel`. | Count "missing" pieces from the FEN, or carry capture data on the wire. | Counting from the FEN is unsound across promotions (a promoted pawn masquerades as a captured pawn); the `useGameState` mirror calls `chess.load`, which discards move history, so it cannot supply captures; replaying the already-validated SAN is exact and needs no protocol change. | A SAN the replay cannot parse would undercount; mitigated by a defensive parse that never throws and by deriving only from server-approved moves. |
| Backend tests use a session-scoped synchronous `TestClient` for both WebSocket and REST, and a function-scoped `httpx.AsyncClient` for REST-only checks. | A function-scoped `TestClient`, or one client for both transports. | The app lifespan registers OpenTelemetry instrumentation and Prometheus collectors once, so re-entering it per test raises duplicate-registration errors; `httpx`'s ASGITransport cannot open a WebSocket scope, so the WebSocket suites need the synchronous client. | Shared client state across a session; mitigated by an autouse room-registry cleanup fixture in the multiplayer suite. |
| Self-play tests inject `AsyncMock` server and recorder seams, scripted searchers, and a no-op sleep, and never launch Playwright, a browser, recording, or the server. | Drive a real browser and server under Playwright inside the test. | Mocked seams keep the suite headless, fast, and deterministic; the runner imports Playwright lazily, so the module imports cleanly without Playwright installed. | Mocks can drift from the real runner; mitigated by asserting the start → record → play → transcript → shutdown call sequence. |
| Evaluator tests assert signs, ranges, monotonic relationships, mirror symmetry, and cache invariances, and cross-check material against `chess_ai.engine.tables.PIECE_VALUES`. | Assert exact piece-square-derived centipawn magnitudes. | Property assertions stay valid when the engine author re-tunes the piece-square tables, whereas exact magnitudes are brittle; cross-checking the engine's own values avoids duplicating tuning data in the tests. | A property test can miss a pure magnitude regression; acceptable because the tuning is expected to change. |
| Suppress `StarletteDeprecationWarning` from `starlette.testclient` with a targeted `filterwarnings` entry in `backend/pyproject.toml`. | Install `httpx2` as Starlette 1.2.1 suggests, or drop the synchronous `TestClient`. | `httpx2` is unavailable offline and the AAP pins `httpx`; the synchronous `TestClient` is required because `httpx` cannot open a WebSocket scope, so the WebSocket suites cannot drop it. The warning subclasses `UserWarning`, so the existing `DeprecationWarning` filter does not catch it. **(deviation)** | A genuine future Starlette break would be hidden; revisit once the `httpx2` pairing is installable. |
| Finalize the self-play recording by transcoding WebM→MP4 with an MP4-capable `ffmpeg` discovered at runtime (a system `ffmpeg` on `PATH` first, then Playwright's bundled binaries), validate the output is a real MP4 (`ftyp` box), and raise `RecordingFinalizationError` when no capable `ffmpeg` exists. | Copy the recorded WebM bytes into the `.mp4` name when `ffmpeg` is absent (the earlier fallback), or assume Playwright's bundled `ffmpeg` can mux MP4. | A `.mp4` that actually holds WebM bytes fails Constraint 14's MP4-artifact requirement; Playwright's bundled `ffmpeg` is built with only the WebM and image muxers (no MP4 muxer or H.264 encoder), so its mere presence is insufficient. Each candidate is capability-probed via `-muxers`, and finalization fails loudly instead of emitting a mislabeled file. **(deviation)** | `make self-play` needs a system `ffmpeg` with `libx264` to produce the MP4 and fails loudly on a host without one; mitigated by an actionable error message and unit tests covering the no-ffmpeg, bundled-only, success, transcode-failure, and invalid-output paths. |
| Emit generic, field-oriented client-facing text for the malformed-message and invalid-JSON `ProtocolError`s in `backend/chess_ai/rooms/protocol.py`, and chain the original exception (`from exc`) for server-side logs. | Echo `str(exc)` from the `TypeError` / `json.JSONDecodeError` (the prior behavior), or strip the error text to a bare code. | The WebSocket transports forward the `ProtocolError` message to the client, so echoing the raw exception disclosed the implementation language and the internal `MoveMessage` dataclass name (CWE-209; QA finding InfoExp-1). Deriving the missing required field names from `dataclasses.fields` and naming the public `type` discriminator keeps the message actionable without leaking interpreter internals, while `from exc` preserves the detail for logs. **(deviation)** | Slightly less detail in the client error; mitigated by naming the exact missing fields and leaving the field-value validators (which report the offending value) unchanged. |
| Pace self-play by holding each rendered position for `SELF_PLAY_MOVE_DELAY_MS` measured from *after* the browser render is dispatched, and never subtract search wall-time from the on-screen hold. | The original formula that subtracted the whole move's elapsed time, from search start `t0` through render, from the delay budget. | Constraint C14 requires every move to stay visible in the real UI for at least 5s; the `t0`-based formula let a long Hard search consume the budget, so a fast book reply could render with a near-zero visible hold (QA Issue 1 measured 64 of 199 intervals under 5s, minimum 0s). Holding from render start guarantees the full hold regardless of search time, and transcript timestamps then reflect the true on-screen cadence. **(deviation)** | Total self-play wall-time grows by the per-move search time; acceptable for a one-off demonstration. |
| Size the `SelfPlayView` board to its container with a `ResizeObserver`-backed width (capped at the 640px board max) and move the board/commentary two-column split from the `md` to the `lg` breakpoint, with `min-w-0` overflow guards. | Keep the fixed 640px board and the `md:flex-row` split that `SelfPlayView` shipped with. | The fixed 640px board overflowed the viewport at 375px (`documentElement.scrollWidth` 656 > 375) and the `md` split clipped the commentary panel at 768px (QA Issue 2); sizing the board to its container and splitting only at `lg` removes the overflow at 375 and 768 while the 640px cap (AAP §0.5.3) and react-chessboard as the sole board renderer (C15) are preserved. | `ResizeObserver` is feature-detected so jsdom tests do not crash; the board never exceeds 640px, so the 1280 and 1920 desktop layouts are unchanged. |
| Render the executive-deck Mermaid node labels as SVG `<text>` (`htmlLabels: false`, set at the top level the Mermaid v11 flowchart-v2 renderer honors) while keeping the original `flowchart LR` layout and the pinned CDN versions. | Switch the diagram to top-to-bottom (`TB`), shrink the Mermaid font, or simplify the node labels. | The HTML `foreignObject` label width rounded ~8px short of the text (129px versus 137px), so `overflow:hidden` clipped the last character of every node at the 1280×720 viewport (QA Issue 3); SVG text is measured via `getBBox` and cannot clip. `TB` was rejected because it pushed the explanatory caption below the 720px fold, and the label-rendering fix is independent of layout direction. | None significant; the reveal.js 5.1.0 / Mermaid 11.4.0 / Lucide 0.460.0 pins are unchanged and the single diagram is verified to fit with no clipping or overflow. |


## Deviations from the earlier specification

The build deliberately departs from the earlier minimal interpretation. Each deviation is intentional:

- `FastAPI` + WebSocket replaces the earlier `Flask`/WSGI HTTP-JSON server, because real-time play and streamed updates need ASGI.
- `python-chess` replaces the hand-written rules engine as the authoritative source of legality, SAN, FEN, draw detection, and Zobrist hashing.
- A hand-built AI engine (evaluation, search, opening book, optional Syzygy endgames) is added, where the earlier spec had none.
- Observability ships in the first release instead of being deferred.

The prompt and the project rules are authoritative, and the repository is greenfield, so these are choices, not conflicts with existing code.
